- This is natural language text, NOT JSON. No { } [ ], no field:value format.

Structure the paragraph exactly in this sequence, separated by commas:
1. Shot type & Camera Angle: Describe the frame setup (e.g., "Medium shot, eye-level angle, static frame").
2. Main Subject(s), Poses, and Actions:

【video_prompt】

Each panel has an existingVideoPrompt from Phase 3 (may be empty).
Task: CONSTRAIN camera and background to static. Allow moderate character actions with slow-motion.

Output format — follow this EXACT structure. Each panel's video_prompt may contain one or more Shot blocks based on duration:

```
[Scene] <environment description - static, no ambient motion>
[Characters] <character 1 name>: <appearance/clothing from character.description> + <current emotional state in ≤5 words>. <character 2 name>: <same>.

Shot 1 (<shotType>, <duration_seconds>s): Static camera, character actions allowed
<Character 1>: "<dialogue from srtSegment (only if present, max 10 words)>"
Action: <specific character micro-actions, speed/fluidity, emotional nuances>
```

Number of Shot blocks per video_prompt is determined by duration:
- duration < 4s  → 1 shot
- duration 4-5s  → 2 shots (same framing and camera angle, split motion progression only — do NOT change angle between shots)
- duration 6-8s  → 3 shots
- duration > 8s  → 4+ shots

Number Shot starting from 1 within each video_prompt entry.

Key rules:
1. [Characters] defines each character ONCE with FULL appearance/clothing from character.description. Do NOT abbreviate — use full description from source data. Then append current emotional state in ≤5 words derived from current.characters[].acting and srtSegment context (e.g. "tense and guarded", "soft and nervous", "calm with hidden grief"). Do NOT invent emotional states — derive only from acting field data.
2. After [Characters] definition, use only the character's NAME in the Shot blocks (Dialogue/Action lines) — the model already knows who they are.
3. The [Scene] section MUST include:
   a. Full environment from current.description and location data — do not summarize.
   b. Environment MUST be described as STATIC — NO ambient motion elements (no leaves rustling, no clouds drifting, no sunlight shifting, no curtain swaying). Background must remain completely still throughout the video.
4. Shot block structure (MANDATORY MULTI-LINE FORMAT): Each Shot must be structured as a multi-line block:
   - Line 1 (Camera & Environment): `Shot X (<shotType>, <duration>s): Static camera` (e.g., "Shot 1 (Medium Shot, 3s): Static camera, environment remains unchanged.")
   - Line 2 (Dialogue - ONLY if srtSegment is present): `<Character Name>: "<short dialogue phrase>"` (translate dialogue to English if input is in Vietnamese/Chinese, keep to ≤10 words)
   - Line 3 (or Line 2 if no dialogue - Action): `Action: <specific character micro-actions, speed/fluidity, emotional nuances>` (e.g., "Action: Bạch Dương pushes the door open and steps into the room, his gaze sweeping forward in a slow, deliberate motion.")
5. Character Actions & Spatial Alignment: Each Action line's action MUST respect spatial positions from image_prompt. If a character is on the left, their action starts from that position.
6. Dialogue in Shot blocks: ALLOWED only when srtSegment is non-empty AND the action involves speaking. Keep to ≤10 words, written exactly on its own line: `Character Name: "Dialogue text"`. Purpose is lip-sync anchor only. Do NOT fabricate dialogue not present in srtSegment. If srtSegment is empty, omit dialogue completely.
7. If existingVideoPrompt exists, enrich it to match this format — add missing [Scene] and [Characters] sections with full detail, add species info, split into correct number of Shot blocks based on duration
8. View Boundary & Camera Scale Constraint (CRITICAL FOR CONTINUITY): Video generation models generate motion using the generated static image as the absolute source.
   - You MUST NEVER zoom out or transition from a tight/close-up/detail shot to a wider shot (e.g., transitioning from a Close-Up of a hand in Shot 1 to a Wide Shot of the room in Shot 2). AI video models cannot maintain facial or clothing consistency when "outpainting" or hallucinating areas not present in the input image.
   - If current.shotType is a Close-Up, Extreme Close-Up, or Detail Shot (e.g., close-up of a prop, hands, or eyes), ALL Shot blocks in the video_prompt must remain strictly locked within that close-up boundary.
   - For close-up panels, only allow: static camera, slow panning/tracking within the close-up, or micro-movements (e.g., fingers twitching, eyes blinking, lips quivering). Do NOT attempt to reveal the face, body, or wider background if they are not visible in the close-up image.
9. Visibility Constraint — CHARACTER LEVEL (MANDATORY): Action for each character in the Action line MUST be strictly restricted to the body parts that are actually VISIBLE in the image_prompt:
   - Hand close-up / Hand visible only   → only hand movements (fingers twitching, tightening, holding)
   - Eye close-up / Eyes visible only    → only eye movements (blinking, pupil dilation, gaze shifts)
   - Blurred back / Background silhouette → static anchor only, body micro-movements (breathing) permitted; NO face/head/limb actions
   - Over-the-shoulder (OTS back visible)→ only subtle shoulder/back shifts or breathing, ABSOLUTELY NO face actions or head rotations
   - Partial/Half-face visible           → only actions/expressions on the visible half of the face
   - Full body visible                   → full body movements, posture changes, walking
   - Full face visible                   → facial expressions, eyes, lips, and head movement
   - NEVER assign actions, rotations, or expressions to body parts that are NOT visible or blurred in the generated static image.

⚠️ **Anti-Jitter / Speed Control Rules** (violation causes stutter, distortion):
- LIMIT ACTIONS per Shot: < 4s → 1 primary action only (a secondary micro-expression is optional but not required). 4-5s → max 2 actions. Never chain 3+ actions in one shot — the AI will rush through them causing jitter
- Keep actions simple and slow: "slowly tilts head", "gently raises hand", "subtle smile" — avoid rapid sequences
- ❌ Bad: "Sonic looks up, waves enthusiastically, then runs away" (too many actions chained = jitter)
- ✅ Good: "Slow-motion. Sonic in a gentle, smooth motion slowly tilts his head upward with a subtle smile. Smooth and fluid movement throughout"
- Use speed keywords in every Shot: "slow-motion", "smooth and fluid", "gentle", "subtle"
- Camera movement: "static camera", "no camera movement" — NEVER use "push", "dolly", "track", "pan", "zoom"

If existingVideoPrompt is NOT empty:
  1. COMPARE Action lines against image_prompt — remove or replace any action that duplicates a motion or state already visible in image_prompt (apply IMAGE → VIDEO CONTINUITY rule above)
  2. RE-VERIFY Shot block count matches the duration rule above — add or remove Shot blocks if needed
  3. Preserve narrative meaning while rewriting actions to satisfy all constraints — do NOT preserve exact actions verbatim
  4. RESTRUCTURE to match the format above if it doesn't already
  5. ENRICH: add missing species from character.description, complete camera movement, specific actions
  6. ABSOLUTELY DO NOT re-describe the static scene already in image_prompt
  7. Always have a "motion" element — no static descriptions

If existingVideoPrompt IS EMPTY → generate from scratch:

  - duration 4-5s: moderate motion (slow head turn, gentle gesture, breathing) — 2 Shot blocks, NO camera movement
  - duration ≥ 6s: moderate motion (standing up, walking slowly, turning body) — 3+ Shot blocks, NO camera movement, keep slow-motion
  - duration null: subtle motion
  - Always use the EXACT format: [Scene] + [Characters] + Shot

【LTX Video 2.3 Specific Guidance】

This pipeline uses LTX Video 2.3 with IC-LoRA (Identity Consistency LoRA) and CropGuides for enhanced character consistency and reduced outpainting.

What LTX 2.3 + IC-LoRA FIXES:
✅ Character face drift (IC-LoRA strength 0.9 maintains identity)
✅ Attribute leakage between characters (IC-LoRA Dual-Character)
✅ Frame expansion (CropGuides crop to visible region)

What LTX 2.3 + IC-LoRA DOES NOT FIX:
❌ Camera push/dolly/track → still causes outpainting at frame edges
❌ Background changes (sunlight shifting, leaves rustling) → still redraws background latent
❌ Character actions requiring invented pixels → still outpaints

Therefore:
- Camera: MUST remain static (NO push, dolly, track, pan, zoom)
- Background: MUST remain static (NO ambient motion elements)
- Character actions: ALLOWED and can be moderate (head turns, gestures, standing up, walking) thanks to IC-LoRA, but MUST use slow-motion and smooth transitions
- Multi-character scenes: Lower risk due to IC-LoRA Dual-Character, but still enforce spatial separation
